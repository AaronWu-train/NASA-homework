\subsubsection{TCP / IP 模型}
\begin{mycolorbox}[teal]
  \footnotesize
  \textbf{Reference:}
  \begin{itemize}
    \item \href{https://blog.zhenghui.idv.tw/2020/07/27/教你如何快速看懂-TCP-IP五層-網路封包/}{教你如何快速看懂 TCP/IP 五層 - 鄭輝的部落格}
    \item \href{https://en.wikipedia.org/wiki/Internet_protocol_suite}{Internet Protocol Suite - Wikipedia}
    \item \href{https://www.ibm.com/docs/en/aix/7.3?topic=protocols-network-interface-packet-headers}{Network interface and packet headers - IBM}
    \item \href{https://ithelp.ithome.com.tw/m/articles/10236217}{TCP/IP 五層結構 - iThome}
    \item 討論對象：chatGPT、喻慈恩
  \end{itemize}
\end{mycolorbox}

TCP / IP 模型總共分為五層，依照由下而上的順序分別是：
\begin{enumerate}
  \item \textbf{Physical Layer}（實體層）：
    \begin{itemize}
      \item 功能：負責網路當中物理上的實體連接，為資料端裝置提供傳送資料的通路，並將數據轉換為電信號或無線電波進行傳輸。
      \item 範例：網路線、無線電波、光纖等。
    \end{itemize}
  \item \textbf{Link Layer}（鏈結層）：
    \begin{itemize}
      \item 功能：定義了在區域網路連結範圍內的設備之間的直接連線的通信方法，會將資料分成 Frame 進行傳輸。主要協議包含 Ethernet、Wi-Fi、PPP 等。這一層具有數據糾錯的功能，可以檢測數據傳輸中的錯誤。
      \item 範例：MAC 位址、交換機（Switch）等。
    \end{itemize}
  \item \textbf{Internet Layer}（網路層）：
    \begin{itemize}
      \item 功能：負責跨網路的數據傳輸，確保數據能夠從來源設備傳送到目標設備。此層會將數據封包透過路由器來決定最佳的傳輸路徑。主要協議包括 IPv4、IPv6、ICMP、ARP 等。
      \item 範例：路由器、IP 位址等。
    \end{itemize}
  \item \textbf{Transport Layer}（傳輸層）：
    \begin{itemize}
      \item 功能：用於不同設備或環境間傳輸訊息，說明如何溝通以及傳遞資料。
      \item 範例：TCP、UDP 等。
    \end{itemize}
  \item \textbf{Application Layer}（應用層）：
    \begin{itemize}
      \item 功能：提供應用程式與網路之間的介面，使應用程式能夠連接網路。
      \item 範例：HTTP（網頁瀏覽）、FTP（檔案傳輸）、SMTP（電子郵件）等。
    \end{itemize}
\end{enumerate}

% Problem 1-2 (a)
\begin{problem}{1-2 (a)} \textbf{虛擬區域網路 (Virtual Local Area Network, VLAN)}
  \begin{itemize}
    \item 功能：VLAN 是一種將網路裝置分割成多個虛擬網段的技術，可以以藉由路由器或是交換器建置，將硬體上在同一個區域網路內的裝置以軟體隔離成不同網段，以達成更有效的管理和資源劃分。
    \item 用途：用來區隔裝置、避免廣播風暴或其他資安危機。
  \end{itemize}
\end{problem}

% Problem 1-2 (b)
\begin{problem}{1-2 (b)} \textbf{Switch v.s. Router}
  \begin{itemize}
    \item 功能與用途：
      \vspace{-0.5em}
      \begin{itemize}
        \item \textbf{Switch}：用於連接多個區域網路（LAN）內的裝置，並且可以根據 MAC 位址來轉發數據，以達到將數據封包傳送到正確的目的地的目的，並減少不必要的流量傳輸。
        \item \textbf{Router}：負責連接不同的網路之間（如 LAN 與 Internet），並且根據 IP 位址決定封包傳輸的最佳路徑，讓設備能夠連接至網際網路或其他網段。同時，支援 NAT 以及 DHCP 等功能，可以將內部網路的私有 IP 位址映射到公共 IP 位址、自動分配內網 IP 位址、管理網路流量等。
      \end{itemize}
    \item 五層架構中扮演的角色：
      \vspace{-0.5em}
      \begin{itemize}
        \item \textbf{Switch}：大部分為第 2 層，在 Link Layer（鏈結層）透過設備 MAC Address 轉發數據。少部分交換器也支援第 3 層交換，可以在網路層根據目的地 IP 位址轉發數據。
        \item \textbf{Router}：屬於第 3 層，在 Internet Layer（網路層）根據 IP 位址轉發數據。
      \end{itemize}
  \end{itemize}
\end{problem}

% Problem 1-2 (c)
\begin{problem}{1-2 (c)} \textbf{Broadcast Storm 和 Switching Loop}
  \begin{enumerate}
    \item \textbf{Broadcast Storm}: Broadcast Storm 指的是在網路當中， Broadcast 跟 Multicast 的封包在網路中不斷的傳播和累積，導致網路流量過大，使得網路效能下降，甚至導致網路癱瘓。Broadcast Storm 的原因可能是因為網路中的裝置不斷的轉發 Broadcast 封包，導致封包在網路中不斷的傳播。

    \item \textbf{Switching Loop}: Switching Loop 是指在網路當中具有大於一條的 layer-2 路徑，例如交換機之間形成一個環。因為在 layer-2 當中是不具有 TTL 機制的，因此封包會在網路中不斷的循環、廣播、擴大，導致 Broadcast Storm 的發生。

    \item 兩者比較：
      \begin{itemize}
        \item Broadcast Storm 是一種網路現象，而 Switching Loop 則是造成 Broadcast Storm 的原因之一。
        \item 除了 Switching Loop 之外，Broadcast Storm 也可能是因為受到 DOS 攻擊、病毒或是錯誤的網路設定等原因導致。
      \end{itemize}
    \item 解決方法：
      \begin{itemize}
        \item \textbf{Broadcast Storm (主要靠流量控制)}:
          \begin{itemize}
            \item 使用 LAN 或是使用 VLAN 來隔離 Broadcast Domain 可以降低 Broadcast Storm 的發生的大小。
            \item 設定 Router 或是防火牆來阻隔攻擊
          \end{itemize}
        \item \textbf{Switching Loop (主要靠網路拓撲規劃與 STP 防範)}:
          \begin{itemize}
            \item 可以啟用 Spanning Tree Protocol（STP）來避免發生：STP 會動態的偵測網路當中是否有 loop 存在，並且自動關閉造成 loop 的連線，將拓樸結構轉為樹狀結構，以避免 Broadcast Storm 的發生。
            \item 妥善配置交換機的連接，避免形成 loop。
          \end{itemize}
      \end{itemize}
  \end{enumerate}
\end{problem}

\subsubsection{IPv4 與 IPv6}
\begin{mycolorbox}[teal]
  \footnotesize
  \textbf{Reference:}
  \begin{itemize}
    \item \href{https://www.ithome.com.tw/tech/92046}{認識IPv4與IPv6的差異 - iThome}
    \item \href{https://aws.amazon.com/tw/compare/the-difference-between-ipv4-and-ipv6/}{IPv4 與 IPv6 有何差異    - AWS}
    \item \href{https://www.koc.com.tw/archives/499314}{IPv4 與 IPv6 兩種網路協定有什麼區別？ - 電腦王阿達}
  \end{itemize}
\end{mycolorbox}
\begin{enumerate}
  \item 為什麼 IPv4 要換成 IPv6？
    \begin{itemize}
      \item 主要原因為 IP 位址不足：IPv4 的 IP 位址總共只有 $2^{32}$ 個，而隨著網路的發展，IP 位址的需求量不斷增加，導致 IP 位址目前已經幾乎用完。IPv6 則有 $2^{128}$ 個 IP 位址，可以解決 IP 位址不足的問題。
    \end{itemize}
  \item 以後有機會從 IPv6 換成其他的嗎？
    \begin{itemize}
      \item IPv6 的位址數量非常大，基本上足以應付未來上百年內的需求了，因此短時間內不太可能會再換成其他的 IP 協議。另外，IPv6 設計時也有保留一定的擴展性，可以應對未來的需求。
    \end{itemize}
  \item IPv4 與 IPv6 的差異？
    \begin{itemize}
      \item IPv4 的 IP 為四個 0 到 255 的數字組成，以小數點隔開。而 IPv6 的 IP 位址則是由八組四位十六進位數字組成，中間以冒號隔開。
      \item 表頭格式：IPv6 的表頭格式有大幅度的修改，讓表頭格式更簡潔並且具有擴充性。
      \item 自動配置：IPv4 需要透過 DHCP 伺服器來分配 IP 位址，不然就要手動分配位址。而 IPv6 使用無狀態地址自動配置 (SLAAC)，讓裝置本身可以在沒有其他設備的情況下自動配置 IP 位址，因此不再需要透過 DHCP 來分配 IP 位址，可以降低網路流量。
      \item 路由效率：IPv6 不需要經由 NAT 來連接網路，因此可以提高路由效率。
      \item 安全性：IPv6 在設計上考量了安全性，例如預設支援 IPsec，以及使用隱私性更強的協議等等，可以提供更好的安全性。
    \end{itemize}
  \item 為什麼還要用 IPv4？
    \begin{itemize}
      \item 目前網路上仍有許多的裝置在使用 IPv4 並且不支援 IPv6，因此全面轉換到 IPv6 需要不少的時間以及大量的成本。因此，現在仍然在使用 IPv4。
    \end{itemize}
\end{enumerate}

\subsubsection{UDP 與 TCP}
\begin{mycolorbox}[teal]
  \footnotesize
  \textbf{Reference:}
  \begin{itemize}
    \item \url{https://ithelp.ithome.com.tw/articles/10205476}
    \item \url{https://www.explainthis.io/zh-hant/swe/tcp-udp}
    \item \url{https://nordvpn.com/zh-tw/blog/tcp-udp-bijiao/}
    \item 討論對象：chatGPT、喻慈恩
  \end{itemize}
\end{mycolorbox}

\begin{enumerate}
  \item \textbf{UDP:}
    \begin{itemize}
      \item 全名：使用者資料包協定 (User Datagram Protocol)
      \item 運作原理：UDP 是一種較簡單的傳輸層協定，傳輸方（Server）不需確認接收方（Client）是否有收到封包，只需要將資料封包傳送出去，而接受端在收到封包之後，也不用回覆回應封包 (ACK) 給發送端。
    \end{itemize}
  \item \textbf{TCP:}
    \begin{itemize}
      \item 全名：傳輸控制協定 (Transmission Control Protocol)
      \item 運作原理：TCP 是一種較為複雜且可靠的傳輸層協定，傳輸方（Server）會透過三次交握確認接收方（Client）是否有收到封包，並且會將封包按照順序傳輸，以確保資料的正確性。
      \item 三次交握：一種確保資料正確性的機制，流程如下：
        \begin{enumerate}
          \item Client 向 Server 主動傳送一個要求連線封包 (SYN)。
          \item Server 收到封包後，回傳一個確認封包 (SYN-ACK)。
          \item Client 收到確認封包後，再回傳一個確認封包 (ACK)，完成三次交握。
        \end{enumerate}
    \end{itemize}
  \item 相同之處：皆為在傳輸層 (layer-4) 的協定，用於在網路上傳輸資料。
  \item 相異之處：

    \vspace{0.5em}
    \renewcommand{\arraystretch}{1.2} % 增加行距，提高可讀性
    \begin{tabular}{|c|c c|}
      \hline
      & \textbf{TCP} & \textbf{UDP} \\
      \hline
      可靠性 & 可靠 & 不可靠 \\
      速度 & 慢，較省資源 & 快，較耗資源 \\
      傳輸方式 & 封包按順序傳輸 & 無法保證順序 \\
      錯誤檢查與修正 & 有 & 無 \\
      壅塞控制 & 有 & 無 \\
      確認 & 有 & 只有檢查碼 \\
      \hline
    \end{tabular}
    \vspace{0.5em}

  \item 何時使用 UDP？何時使用 TCP？：
    \begin{itemize}
      \item UDP：適合用於即時性要求高、傳輸量大、傳輸過程中不需要確認的應用，例如：串流服務、網路遊戲、視訊會議、DNS 查詢、DHCP 等等。
      \item TCP：適合用於需要確認接收方是否有收到封包、傳輸過程中需要保證資料正確性的應用，例如：HTTP (網頁)、FTP (檔案傳輸)、SMTP (電子郵件)、SSH、VPN、線上銀行等等。
    \end{itemize}
\end{enumerate}

  \end{itemize}
  其中，根據 IPv4 的私有 IP 定義 (RFC 1918)，私有 IP 地址範圍如下：

  \begin{itemize}
    \item Class A: \texttt{10.0.0.0 -- 10.255.255.255}
    \item Class B: \texttt{172.16.0.0 -- 172.31.255.255}
    \item Class C: \texttt{192.168.0.0 -- 192.168.255.255}
  \end{itemize}
  因此，\texttt{10.200.200.200} 是一個 Private Network 的 IP 位址。

  另外，若我在瀏覽器當中輸入 \texttt{10.200.200.200}，則會出現台大 sslvpn 登入頁面的畫面（如 \autoref{fig:2-1-c} 所示），但若關閉 VPN 連線後，則無法連線至該 IP 位址。有此可以推知該 IP 位址為位於台大內網的 sslvpn 的伺服器 IP 位址，屬於 Private Network。

  \end{figure}

  至於另外兩個 IP 位址，皆不在私有 IP 地址範圍內，因此可以認定為公有 IP 地址。

\begin{problem}{8.1} \textbf{解讀古老卷軸 (3pt)}
  \begin{itemize}
    \item Dockerfile 內容如以下以及 \autoref{fig:8-1} 所示
      \begin{minted}[frame=lines, linenos, escapeinside=||, framesep=2mm]{Dockerfile}
FROM python:3.9-slim
RUN apt-get update && apt-get install -y \
    build-essential \
    libssl-dev \
    net-tools \
    iproute2 \
    tcpdump \
    tshark \
    nano \
    curl \
    wget \
    vim \
    less \
    procps \
    lsof \
    iputils-ping \
    && rm -rf /var/lib/apt/lists/*

RUN mkdir -p /usr/libexec/run

COPY usr/libexec/run/dist/transfer /usr/libexec/run/transfer
COPY usr/libexec/run/run.sh /usr/libexec/run/run.sh

RUN chmod +x /usr/libexec/run/transfer
RUN chmod +x /usr/libexec/run/run.sh

CMD ["/usr/libexec/run/run.sh"]
      \end{minted}

    \item 第 1 行：\texttt{FROM python:3.9-slim} 指令表示這個 Dockerfile 是基於 \texttt{python:3.9-slim} 映像建立的。
      這是一個精簡版的 Python 3.9 映像，基於 Debian，保留 Python 及必要的系統元件，但刪除了部分工具來減少映像大小。

    \item 第 2 - 16 行：\texttt{RUN} 代表在 Docker 映像建構（build）時 執行的指令。這裡透過 \texttt{apt-get} 安裝了一些必要的 Linux 套件：
      \begin{itemize}
        \item \texttt{build-essential}：編譯 C/C++ 必備的工具（如 \texttt{gcc}）。
        \item \texttt{libssl-dev}：SSL 加密函式庫。
        \item \texttt{net-tools}：網路工具（包含 \texttt{ifconfig}, \texttt{netstat}）。
        \item \texttt{iproute2}：網路工具（包含 \texttt{ip}, \texttt{ss}）。
        \item \texttt{tcpdump}、\texttt{tshark}：網路封包擷取與分析
        \item \texttt{nano}、\texttt{vim}：文字編輯器。
        \item \texttt{curl}、\texttt{wget}：HTTP 下載工具。
        \item \texttt{less}：文字檢視工具，適合瀏覽長文件。
        \item \texttt{procps}：包含 \texttt{ps}, \texttt{top}, \texttt{kill} 等指令，用於系統進程管理。
        \item \texttt{lsof}：列出開啟的檔案，適合偵測哪些程序正在存取特定檔案或網路連線。
        \item \texttt{iputils-ping}：提供 \texttt{ping} 指令，測試網路連線。
      \end{itemize}

    \item 第 17 行：\texttt{rm -rf /var/lib/apt/lists/*} 這行指令清除 \texttt{apt} 快取，以減少 Docker 映像的大小。

    \item 第 19 行：\texttt{mkdir -p /usr/libexec/run} 確保 \texttt{/usr/libexec/run} 目錄存在，用於存放後續執行的腳本或工具。

    \item 第 21 - 22 行：\texttt{COPY} 指令將檔案從 Host 複製到容器內相對應的位置。

    \item 第 24 - 25 行：\texttt{chmod +x} 指令賦予\texttt{transfer} 和 \texttt{run.sh} 執行權限：
    \item 第 27 行：\texttt{CMD} 可以指定容器啟動時的預設指令。當容器啟動時，會執行 \texttt{run.sh} 腳本。
  \end{itemize}
