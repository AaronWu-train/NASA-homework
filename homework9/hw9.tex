\setcounter{subsection}{-1}
\subsection{Setup}
\begin{enumerate}
  \item 首先，我啟動三台虛擬機之後，以 VNC 連線上去，並且以 lab 的方法啟動 \texttt{ens3} 與 \texttt{ens4} 兩個網路介面卡。
  \item 虛擬機中沒有 \texttt{dhclient}，所以我建立 \texttt{/etc/netplan/nasa.yaml} 來手動設定 \texttt{ens3} 和 \texttt{ens4} 的IP 位址。
  \begin{minted}{yaml}
    network:
      ethernets:
          ens3:
              addresses: [10.0.2.15/24]
              nameservers:
                  addresses: [8.8.8.8]
              routes:
              -   to: default
                  via: 10.0.2.2
          ens4:
              addresses: [192.168.128.73/16]
      version: 2
  \end{minted}
  \item 將其他的 yaml 檔案移除，並且執行 \texttt{sudo netplan apply} 來套用設定。
  \item 設定完成後，三台虛擬機的 IP 位址分別為：
  \begin{itemize}
    \item \texttt{hpc1}: \texttt{192.168.128.71}
    \item \texttt{hpc2}: \texttt{192.168.128.72}
    \item \texttt{nfs-server}: \texttt{192.168.128.73}
  \end{itemize}
\end{enumerate}

\subsection{NFS Server 端設定}
\begin{enumerate}
  \item 我按照 lab 的指示，安裝 NFS Server，並編輯 \texttt{/etc/exports} 檔案，加入以下內容：
  \begin{minted}{bash}
    /srv/nfs-share 192.168.128.0/16(rw,sync,fsid=0,crossmnt,no_subtree_check) 
  \end{minted}
  \item 更改資料夾權限：
  \begin{minted}{bash}
    sudo mkdir -p /srv/nfs-share
    sudo chown 1000:1000 /srv/nfs-share/
    sudo chmod 777 /srv/nfs-share
  \end{minted}
  \item 套用設定並且啟動 NFS Server：
  \begin{minted}{bash}
    sudo exportfs -arv 
    sudo systemctl enable --now nfs-server
    sudo systemctl restart nfs-server
  \end{minted}

\section{效能與大規模檔案測試}

\begin{mycolorbox}[teal]
  \textbf{Reference:}
  \footnotesize
  \begin{itemize}
    \item \url{https://tldp.org/HOWTO/NFS-HOWTO/performance.html}
    \item \href{https://unix.stackexchange.com/questions/559438/ssh-multiple-hosts-with-respective-users}{ssh multiple hosts with respective users}
    \item \href{https://shazi.info/ssh-%E7%94%A8-sshpass-%E5%8F%96%E4%BB%A3-expect-%E8%87%AA%E5%8B%95%E8%BC%B8%E5%85%A5%E5%AF%86%E7%A2%BC/}{ssh 用 sshpass 取代 expect 自動輸入密碼}
    \item \href{https://www.server-world.info/en/note?os=Ubuntu_20.04&p=ssh&f=10}{OpenSSH : Use Parallel SSH}
  \end{itemize}
\end{mycolorbox}
\begin{enumerate}
  \item 首先，為了同步化測試，我在 server 上安裝 \texttt{parallel-ssh} 和 \texttt{sshpass} 套件。
  \begin{minted}{bash}
    sudo apt install pssh sshpass
  \end{minted}
  \item 由於 server 空間不夠，因此我用以下指令擴充 lvm 的空間：
  \begin{minted}{bash}
    sudo lvextend -l +100%FREE ubuntu-vg/ubuntu-lv
    sudo resize2fs /dev/mapper/ubuntu--vg-ubuntu--lv
  \end{minted}
  \item 我可以透過編輯 server 的 \texttt{/etc/exports} 從 \texttt{sync} 模式切換到 \texttt{async} 模式：
  \begin{minted}{bash}
    /srv/nfs-share 192.168.128.0/16(rw,async,fsid=0,crossmnt,no_subtree_check,root_squash)
  \end{minted}
  完成後，使用以下指令套用設定：
  \begin{minted}{bash}
    sudo exportfs -arv 
    sudo systemctl restart nfs-server
  \end{minted}

  \item 在 \texttt{hpc1} 與 \texttt{hpc2} 上，我可以用不同 mount 參數來修改 \texttt{rsize} 和 \texttt{wsize} 的大小：
  \begin{minted}{bash}
    sudo umount /mnt/nfs-share
    sudo mount -t nfs -o rsize=32768,wsize=32768 192.168.128.73:/srv/nfs-share /mnt/nfs-share
  \end{minted}
  \item 我建立了一個 \texttt{hosts.txt} 的 host 清單，內容如下，並放在 nfs 共享目錄中：
  \begin{minted}{text}
    astro1@192.168.128.71
    astro2@192.168.128.71
    astro3@192.168.128.71
    astro1@192.168.128.72
    astro2@192.168.128.72
    astro3@192.168.128.72
  \end{minted}
  

  \item 我撰寫的 shell script \texttt{nfs\_test.sh} 會依序做以下事情：
  \begin{enumerate}
    \item 根據不同參數，決定使用 \texttt{single} 或 \texttt{parallel} 的方式來執行測試。若要測試 \texttt{single}，可以直接在 \texttt{hpc1} 或 \texttt{hpc2} 上執行 \texttt{nfs\_test.sh}。若要測試 \texttt{parallel}，則需在 server 上執行，並給予 host 清單以及輸入密碼，使得我的 shell script 可以透過 \texttt{parallel-ssh} 自動登入 \texttt{hpc1} 和 \texttt{hpc2}。
    \item 我使用 \texttt{dd} 指令來將一個大檔案寫入 NFS Server 的資料夾中。其中，\texttt{iflag=direct} 和 \texttt{oflag=direct} 參數可以讓 \texttt{dd} 指令直接寫入磁碟，而不經過快取，這樣可以更真實地測試 NFS 的效能。
    同時，\texttt{dd} 指令會將寫入的時間和傳輸速率等資訊輸出。
    \item 我使用 \texttt{/usr/bin/time -v} 指令來獲得測試的 CPU 效能資訊，並將結果寫入檔案。
  \end{enumerate}

\subsection*{問答題：}
在多使用者的情境下，可以看到花費時間顯著上升，且傳輸速率下降。 由於我們在 \texttt{sync} 和 \texttt{async} 模式下所花費的時間並沒有顯著差異。另外，從 CPU 負載的數據來看，當使用者數量增加時，CPU 負載並沒有顯著上升，這意味著 CPU 並不是造成性能下降的主要原因。因此我們可以推斷造成瓶頸的位置為網路頻寬不足，導致讀寫速度下降。要解決此問題的話，可以考慮使用更高效的網路介面卡，或著是將連接 \texttt{NFS Server} 的網路頻寬提升來提升性能。同時，盡量保持較大的 \texttt{rsize} 和 \texttt{wsize} 也是一個提升傳輸速率的好方法。
