\subsection{SSRF、CSRF 和 XSS}
\begin{itemize}
  \item \textbf{SSRF（Server-Side Request Forgery）:}

  在這種攻擊方式中，攻擊者利用伺服器端的向特定的另一個服務發送請求，通常是為了繞過防火牆或存取內部網路資源。常見的情況是，假如你的網站上有個輸入框，讓使用者自己填網址。攻擊者就可以偷偷填一個公司內網的地址，讓你的伺服器去「自動」幫他連過去。結果可能拿到內網機密、撈到不該給外部看的資料，甚至繞過防火牆去搞其他事。

  \item \textbf{CSRF（Cross-Site Request Forgery）:}

  在這種攻擊中，攻擊者誘使使用者在已經登入的情況下，透過攻擊者先在別的網站或郵件裡夾一個隱藏的請求連結，發送未經授權的請求到另一個網站。只要你正好在同一瀏覽器裡已經登入了某個網站，瀏覽器就會帶上你的 Cookie／驗證資訊，自動幫你送出那個請求，並用你的身份執行操作。結果你可能就被偷轉帳、改設定或亂發文。

  \item \textbf{XSS（Cross-Site Scripting）:}
  
  基本上，攻擊者把惡意 JavaScript 或 HTML 混進你信任的網頁裡，等人一打開網頁，腳本就在使用者的瀏覽器跑起來。可以偷 Cookie、竊取表單資料，甚至直接在頁面上做釣魚操作。

  \item \textbf{三者比較:} 
  \begin{table}[H]
    \centering
    \begin{tabular}{p{2cm}ccc}
      \toprule
      \textbf{攻擊類型} & \textbf{SSRF} & \textbf{CSRF} & \textbf{XSS} \\
      \midrule
      \textbf{攻擊對象} & 伺服器 & 使用者 & 使用者 \\
      \textbf{攻擊方式} & 利用伺服器發送請求 & 誘使使用者發送請求 & 注入惡意腳本 \\
      \textbf{影響範圍} & 伺服器內部資源 & 使用者帳號 & 使用者瀏覽器 \\
      \bottomrule
    \end{tabular}
    \caption{SSRF、CSRF 和 XSS 的比較}
    \label{tab:ssrf-csrf-xss}
  \end{table}
\end{itemize}

\subsection{漏洞分析}
\subsubsection*{Vulnerability 1: Double Free}
PoC \texttt{crashes/id:000003,sig:06,src:000591,time:100031,execs:122095,op:havoc,rep:16}\\會導致程式觸發 double free 的漏洞。當我在 gdb 中配合 AddressSanitizer 執行該 PoC 時，會發現在 \texttt{fuzz-me.c:6655:5} 處觸發了 double free 的錯誤。

根據 \texttt{gdb} 的分析結果，可以發現在函數 \texttt{mlxeotckigrz} 當中，陣列 \texttt{cghjfucyoytm} 的內容在 \texttt{fuzz-me.c:6641:51} 中被宣告，並且在 \texttt{fuzz-me.c:6652:9} 當中，在進入 if 區塊中被釋放了第一次。然而出來到第 6655 行時，程式又無條件釋放了同一個陣列，導致 double free 的錯誤。

\subsubsection*{Vulnerability 2: Stack Buffer Overflow}
PoC \texttt{crashes/id:000011,sig:06,src:000488,time:196159,execs:241438,op:havoc,rep:14}\\會導致程式 stack buffer overflow。當我在 gdb 中配合 AddressSanitizer 執行該 PoC 時，會發現 stack overflow 的觸發地點在 \texttt{fuzz-me.c:3040:9} 處。

我檢查該處的程式，可以發現第 3040 行位於 \texttt{ngbaogfcgfjv} 函數當中，該行內容為透過 \texttt{memcpy} 進行記憶體複製，從 \texttt{rvmltrxnyedr} 將長度 \texttt{nhtdbsyeyipt} 的記憶體複製到 \texttt{qolxxudqisef}。
\begin{minted}{c}
  memcpy(qolxxudqisef, rvmltrxnyedr, nhtdbsyeyipt);
\end{minted}
然而，根據第 3039 行可以看到 \texttt{nhtdbsyeyipt} 的值為 $\le 510$ 的整數，而第 3029 行當中宣告 \texttt{qolxxudqisef} 的長度為 114 字元，因此當 \texttt{nhtdbsyeyipt} 的值大於 114 時，就會導致 stack buffer overflow。

\subsubsection*{Vulnerability 3: Segmentation fault}
PoC \texttt{crashes/id:000020,sig:11,src:000802,time:331908,execs:354578,op:havoc,rep:2}\\ 會導致程式觸發 segmentation fault。當我在 gdb 中執行該 PoC 時，會發現在 \texttt{fuzz-me.c:3022} 處，\texttt{if (*kfdujeztzdrf > 0)} 的條件判斷式中，因為指標\texttt{kfdujeztzdrf} 存取到沒被分配的記憶體位置，導致程式觸發 segmentation fault。

\subsection{猜數字}
\textit{Steps:}
\begin{enumerate}
  \item 首先，我觀察了 \texttt{fatcat.py} 的程式碼，發現它使用了一個線性同餘生成器（LCG）來產生隨機數。這個生成器的公式為：
  \begin{equation*}
    X_{n+1} = (a \cdot X_n + c) \pmod m
  \end{equation*}
  \item 在程式碼當中我們已知 m ，但是並不知道 a 和 c 的值。然而，根據我搜尋的結果，LCG 演算法的 a 和 c 的值可以透過測量三次隨機數的差值來計算出來。過程如下：
  \begin{equation*}
    \begin{aligned}
      X_1 &\equiv a\,X_0 + c \pmod{m},\\
      X_2 &\equiv a\,X_1 + c \pmod{m},\\
      X_2 - X_1 &\equiv a\,(X_1 - X_0) \pmod{m}
    \end{aligned}
    \quad\Longrightarrow\quad
    \begin{aligned}
      a &\equiv \frac{X_2 - X_1}{X_1 - X_0} \pmod{m},\\
      c &\equiv X_1 - a\,X_0 \pmod{m}.
    \end{aligned}
  \end{equation*}
  \item 我根據和伺服器互動三次，取得了 $X_0$、$X_1$ 和 $X_2$ 的值。接著，我透過 \texttt{solve\_lcg.py} 程式碼計算出 a 和 c 的值。
  \item 我撰寫了 \texttt{predict\_lcg.py} 程式碼，並且在裡面透過 \texttt{pwntools} 套件來和伺服器互動，並且使用計算出來的 a 和 c 的值來預測下一個隨機數。猜中答案 100 次後，我取得足夠多信任值，並且成功獲得了 flag。
\end{enumerate}

\subsection{考試答案卷}
\textit{Steps:}
\begin{enumerate}
  \item 觀察程式，可以看到產生 flag 2 加密的程式如下：
  \small
  \begin{minted}{python3}
    def OTPEncrypt(msg: bytes) -> bytes:
        key_len = 10
        key = secrets.token_bytes(key_len)
        enc = bytes(msg[i] ^ key[i % key_len] for i in range(len(msg)))
        return enc

    def send_exam_result():
        hexlify(OTPEncrypt(FLAG2.encode())).decode()
  \end{minted}
  \normalsize
  這個加密方式是將一組長度為 10 的位元組陣列作為金鑰，重複循環的與明文逐位做 XOR 運算來加密。由於明文中一定包含前綴 \texttt{NASA\_HW11\{}，而密文 XOR 明文正好會還原出金鑰，因此可利用這段已知字串來破解整個金鑰。
